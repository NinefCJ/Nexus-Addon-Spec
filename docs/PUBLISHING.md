## 版本号规范

### 语义化版本

Nexus 拓展包使用 [语义化版本 2.0.0](https://semver.org/lang/zh-CN/)：

```
主版本号.次版本号.修订号
   MAJOR.MINOR.PATCH
```

| 版本号 | 说明 | 示例 |
|--------|------|------|
| **主版本 (MAJOR)** | 不兼容的变更，用户需要注意 | 1.0.0 → 2.0.0 |
| **次版本 (MINOR)** | 向下兼容的新增功能 | 1.0.0 → 1.1.0 |
| **修订号 (PATCH)** | 向下兼容的问题修复 | 1.0.0 → 1.0.1 |

### 详细规则

#### 修订号 (PATCH) 递增

以下情况递增修订号：

- 🔤 修正文字拼写错误
- 📝 修正描述不准确
- 🐛 修复 ID 命名错误
- 🎨 调整分类名称
- 📐 修正命令参数错误

**示例：**
```
1.0.0 → 1.0.1  # 修正了一个物品的描述文字
1.2.3 → 1.2.4  # 修复了一条命令的语法错误
```

#### 次版本号 (MINOR) 递增

以下情况递增次版本号：

- ✨ 新增方块/物品/音效/粒子
- ⚡ 新增快捷命令
- 📋 新增命令模板
- 🏷️ 新增分类
- ⚡ 功能增强（不影响兼容性）

**示例：**
```
1.0.0 → 1.1.0  # 新增 5 个自定义物品
1.2.3 → 1.3.0  # 新增命令模板分类
```

#### 主版本号 (MAJOR) 递增

以下情况递增主版本号：

- 💥 大量删除原有内容
- 🔄 大规模重构 ID 命名
- 📦 拆分为多个拓展包
- 🚫 放弃兼容旧版本 Nexus

**示例：**
```
1.5.2 → 2.0.0  # 大规模重命名所有 ID
```

### 特殊版本标记

#### 预发布版本（可选）

如果是测试版，可以在版本号后加上预发布标记：

```
1.0.0-alpha    # 内部测试版
1.0.0-beta     # 公开测试版
1.0.0-rc.1     # 发布候选版
```

> 💡 建议普通用户直接使用正式版。预发布版本适合愿意尝鲜的用户。

## 发布流程

### 完整发布流程图

```
开发完成
   ↓
自测通过 ✅
   ↓
更新版本号
   ↓
编写更新日志
   ↓
最终检查
   ↓
发布
   ↓
公告宣传
```

### 详细步骤

#### Step 1: 完成开发

确保所有计划中的功能都已实现：

- [ ] 所有内容已添加完毕
- [ ] 所有 ID 命名规范
- [ ] 描述信息完整准确

#### Step 2: 全面自测

按照 [调试与测试指南](./DEBUGGING.md) 中的测试清单逐项检查。

#### Step 3: 确定版本号

根据变更内容确定新版本号：

| 变更类型 | 版本变化 |
|---------|---------|
| 只修复了 Bug / 修正文字 | PATCH +1 |
| 新增了内容 / 功能 | MINOR +1 |
| 大规模变更 / 不兼容 | MAJOR +1 |

#### Step 4: 更新 JSON 中的版本号

修改 `addon.json` 中的 `version` 字段：

```json
{
  "id": "your.addon",
  "version": "1.1.0",   // ← 更新这里
  ...
}
```

#### Step 5: 编写更新日志

记录本次版本的变更内容。详见 [更新日志编写](#更新日志编写)。

#### Step 6: 最终检查

发布前最后确认一遍：

- [ ] 版本号已正确更新
- [ ] 更新日志已编写
- [ ] JSON 格式正确
- [ ] 所有功能测试通过
- [ ] 没有包含敏感或侵权内容

#### Step 7: 发布

选择合适的方式发布：

- **GitHub Release**（推荐）
- **社区帖子**
- **直接分享文件**

#### Step 8: 宣传

告诉用户新版本发布了：

- 更新 README
- 发布公告
- 回复用户反馈

## 发布前检查清单

### 内容检查

- [ ] 所有计划的内容都已添加
- [ ] 没有遗留的 TODO 或占位符
- [ ] 所有描述准确无误
- [ ] 分类名称一致，没有拼写错误
- [ ] 没有空的数组或对象（除非有意为之）

### 规范检查

- [ ] ID 命名符合规范（全小写、下划线、独特命名空间）
- [ ] 没有使用 `minecraft:` 前缀的自定义内容
- [ ] 所有命令都以 `/` 开头
- [ ] 版本号符合语义化规范
- [ ] JSON 格式正确，能通过校验

### 功能检查

- [ ] 方块补全正常
- [ ] 物品补全正常
- [ ] 音效补全正常
- [ ] 粒子补全正常
- [ ] 快捷命令正常
- [ ] 命令模板正常
- [ ] 启用/禁用正常
- [ ] 卸载正常

### 文档检查

- [ ] 版本号已更新
- [ ] 更新日志已编写
- [ ] README 已更新（如果有）
- [ ] 说明文档完整

## 更新日志编写

### 为什么需要更新日志？

- 用户知道新版本有什么变化
- 出现问题时可以追溯版本
- 体现项目的活跃度和维护情况

### 更新日志格式

推荐使用 **Keep a Changelog** 格式：

```markdown
# 更新日志

## [1.1.0] - 2026-06-27

### 新增
- 新增 5 个自定义武器
- 新增建筑类命令模板
- 新增红石音效

### 修复
- 修正了「魔法剑」的描述文字
- 修复了传送命令的坐标错误

### 变更
- 调整了部分物品的分类
```

### 常用分类标签

| 标签 | 说明 |
|------|------|
| **新增** | 新功能、新内容 |
| **修复** | Bug 修复 |
| **变更** | 现有功能的调整 |
| **废弃** | 即将移除的内容 |
| **移除** | 已移除的内容 |
| **安全** | 安全相关的修复 |

### 好的更新日志示例

```markdown
## [2.0.0] - 2026-06-20

### ⚠️ 重要提示
本次更新为大版本更新，部分内容 ID 有变化。如果之前收藏了相关命令，可能需要重新添加。

### 新增
- 新增 15 个装饰方块
- 新增 8 个工具类快捷命令
- 新增「魔法物品」分类

### 修复
- 修正了「烈焰之剑」的伤害值描述
- 修复了 3 条命令的语法错误

### 变更
- 统一了所有物品的 ID 命名
- 重新整理了分类结构

### 移除
- 移除了过时的「旧版武器」系列
```

## GitHub 发布指南

### 推荐仓库结构

```
my-nexus-addon/
├── addon.json              # 拓展包主文件
├── README.md               # 说明文档
├── CHANGELOG.md            # 更新日志
├── LICENSE                 # 开源协议（可选）
└── preview/
    └── showcase.png        # 预览图（可选）
```

### README 模板

```markdown
# 拓展包名称

一句话描述这个拓展包是做什么的。

## ✨ 功能特性

- 特性 1
- 特性 2
- 特性 3

## 📦 内容统计

| 类型 | 数量 |
|------|------|
| 方块 | 0 |
| 物品 | 0 |
| 音效 | 0 |
| 粒子 | 0 |
| 快捷命令 | 0 |
| 命令模板 | 0 |

## 🚀 安装方法

1. 下载 `addon.json` 文件
2. 打开 Nexus 命令助手
3. 进入「设置」→「拓展包管理」
4. 点击右上角「+」按钮
5. 粘贴 JSON 内容，点击导入
6. 开启拓展包即可使用

## 📝 更新日志

详见 [CHANGELOG.md](./CHANGELOG.md)

## 📄 许可证

MIT License（或其他协议）
```

### GitHub Release 发布步骤

1. **进入仓库主页**，点击右侧「Releases」
2. **点击「Draft a new release」**
3. **填写版本信息**：
   - **Tag version**: `v1.0.0`（版本号前加 v）
   - **Release title**: `v1.0.0 - 初始版本`
   - **Describe this release**: 粘贴更新日志
4. **上传文件**：将 `addon.json` 拖进去
5. **点击「Publish release」**

## 更新维护策略

### 多久更新一次？

没有固定的频率，根据实际情况：

| 更新类型 | 建议频率 |
|---------|---------|
| 修复 Bug | 发现后尽快发布 |
| 小量新增内容 | 每周/每两周 |
| 大量新内容 | 每月/每季度 |

### 版本兼容

- **MINOR 和 PATCH 更新**：用户直接升级即可，无缝衔接
- **MAJOR 更新**：建议提前通知用户，说明迁移方法

### 旧版本支持

- 至少维护最新一个主版本的 Bug 修复
- 旧版本如果有严重问题，可以考虑发补丁

### 用户反馈处理

- 收到反馈后及时回复
- 确认是 Bug 的话安排修复
- 好的建议可以加入后续版本计划

## 废弃与下架

### 什么时候考虑废弃？

- 内容已经过时不再适用
- 有了更好的替代方案
- 维护者不再有时间维护

### 废弃流程

1. **发布公告**：说明即将废弃的原因和时间
2. **过渡期**：至少保留一个月的过渡期
3. **最终版本**：发布一个标记为废弃的版本
4. **归档**：将仓库标记为 Archived（如果在 GitHub）

### 废弃版本的标记

可以在描述中加上废弃标记：

```json
{
  "id": "your.old_addon",
  "name": "【已废弃】旧拓展包",
  "description": "此拓展包已不再维护，请使用 xxx 替代",
  ...
}
```

## 推荐的发布工作流

### 小版本（PATCH）快速发布

```
发现 Bug
   ↓
修复并自测
   ↓
递增修订号
   ↓
简单的更新日志
   ↓
直接发布
```

### 中版本（MINOR）标准发布

```
功能开发完成
   ↓
完整自测
   ↓
递增次版本号
   ↓
编写详细更新日志
   ↓
最终检查
   ↓
发布
   ↓
发布公告
```

### 大版本（MAJOR）慎重发布

```
大规模重构 / 大量变更
   ↓
全面测试
   ↓
内部测试版（alpha/beta）
   ↓
收集反馈并调整
   ↓
发布候选版（RC）
   ↓
最终确认
   ↓
正式发布
   ↓
详细的迁移指南
```
